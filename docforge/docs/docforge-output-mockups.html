1.Executive Summary

The Fairview environment is a single-forest, single-domain Active Directory deployment (corp.fairviewcity.gov) spanning 3 physical sites connected by dedicated fiber and a site-to-site VPN. Two writable domain controllers serve the primary datacenter, with one additional DC at the Operations Center for site resilience. DNS is AD-integrated; DHCP is centralized with failover configured between the two datacenter DCs.

2.User Account Management

SOP: Create a standard user

  1. Confirm an approved onboarding ticket exists (HR feed or ServiceDesk).
  2. Create the account in the department OU — e.g. OU=Public Works,OU=Staff,DC=corp,DC=fairviewcity,DC=gov. Never create users in the default Users container; no departmental GPOs apply there.
  3. Username standard observed in this environment: first initial + last name (e.g. jrivera); collisions append a digit.
  4. Add to the department baseline group (SG-Dept-PublicWorks) — file/print and VPN access flow from group membership only.
  5. Set User must change password at next logon; deliver the temporary password out-of-band.
# Runbook-standard creation (values from ticket)
New-ADUser -Name "Jordan Rivera" -SamAccountName jrivera `
  -UserPrincipalName jrivera@corp.fairviewcity.gov `
  -Path "OU=Public Works,OU=Staff,DC=corp,DC=fairviewcity,DC=gov" `
  -AccountPassword (Read-Host -AsSecureString) `
  -ChangePasswordAtLogon $true -Enabled $true
Add-ADGroupMember SG-Dept-PublicWorks -Members jrivera

SOP: Disable on separation

  1. Disable the account and move it to OU=Disabled,OU=Staff,… (GPO there strips logon rights).
  2. Record ticket # and date in the description attribute.
  3. Retain 90 days, then delete. Litigation-hold accounts are flagged by the City Attorney's office before deletion.

Disaster Recovery Plan — Identity & Core Network Services
City of Fairview · corp.fairviewcity.gov · Generated 2026-07-07 · DocForge v1.0

1.Executive Summary

This plan covers recovery of Active Directory, DNS, DHCP, and endpoint management (MECM) for the City of Fairview. The environment's two-DC HQ core with one remote DC supports a 4-hour RTO for authentication services; the primary risks are a full HQ site loss (all FSMO roles co-located) and the DC-less Corporation Yard site.

3.Critical Infrastructure Inventory

  • FVDC01 / FVDC02 — HQ; hold all 5 FSMO roles between them; DNS + DHCP failover pair.
  • FVDC03 — Ops Center; GC + DNS; the only identity asset outside HQ.
  • FVSCCM01 — MECM primary (FV1); SQL local; content library 1.8 TB.
  • Backups: nightly system-state of all DCs to \\fvbackup01\adsysstate$, weekly offsite copy.

4.Risk Assessment — Single Points of Failure

5.Recovery Procedures

Scenario A — Single DC failure (FVDC01)

  1. Confirm outage; verify FVDC02/FVDC03 healthy: dcdiag /e /q.
  2. Seize FSMO roles held by the failed DC to FVDC02 (see runbook §6).
  3. Point DHCP failover authoritative to survivor; verify scopes serving.
  4. Metadata-cleanup the failed DC (ntdsutil or ADUC delete) before rebuilding under the same name only after cleanup replicates.

Scenario B — Complete HQ site loss

  1. FVDC03 continues authentication for Ops Center immediately; CorpYard rides the VPN re-route.
  2. Seize all five FSMO roles to FVDC03.
  3. Restore DHCP database from nightly backup onto FVDC03; recreate failover later.
  4. Stand up replacement DCs at the recovery site; MECM restore proceeds per §RTO (72 h).

Scenario C — AD database corruption (USN rollback / lingering objects)

  1. Isolate the suspect DC from the network immediately.
  2. Do not snapshot-revert a DC. Restore from system-state backup using wbadmin start systemstaterecovery in DSRM.
  3. Mark restore authoritative only for deliberately deleted objects (ntdsutil "authoritative restore").

Scenario D — DNS/DHCP service failure

DNS: any surviving DC serves the zone; repoint DHCP option 006 if a server is lost. DHCP: activate failover partner or restore dhcp.mdb from backup; leases survive up to their duration.

Scenario E — Inter-site connectivity loss

Ops Center: fully self-sufficient (FVDC03). Corporation Yard: cached credentials only — communicate the 10-hour cached-logon window to site supervisors; escalate to carrier per Communication Plan.

6.Backup Requirements

  • System-state on ≥2 DCs nightly, retained 30 days, weekly copy offsite — currently met.
  • DHCP database nightly (netsh dhcp server backup) — currently met, restore untested.
  • MECM: SQL full nightly + CD.Latest folder; content library excluded (redistributable).

1.Executive Summary

Overall risk rating: High

The environment shows solid fundamentals — a modern password policy, enforced BitLocker, and 94% patch compliance — but identity hygiene drags the rating down: 47 accounts carry non-expiring passwords, 62 users and 71 computers are stale beyond 90 days, and two legacy Server 2012 R2 hosts remain domain-joined past end of support. 3.Detailed Findings

F-01 · Privileged service accounts Critical

Three service accounts (svc-backup, svc-tyler, svc-legacyapp) are members of Domain Admins with passwords set to never expire; svc-legacyapp's password age is 1,847 days. Any credential theft on a member server escalates directly to domain compromise.

Remediate by moving each to least-privilege delegation or gMSA; where the vendor requires local admin, scope it per-server, never at the domain level.

F-02 · Legacy operating systems High

FVAPP03 and FVPRINT01 run Server 2012 R2, unsupported since Oct 2023 and outside the patch compliance figures below. Both sit on the server VLAN with unrestricted east-west reachability.

F-03 / F-04 · Identity hygiene High Med

44 standard users hold non-expiring passwords (largely a 2019-era exemption group that was never revisited) and 62 enabled users have not logged on in 90+ days — including 11 in the separated-employees pattern that were disabled in HR but not in AD. Stale computer objects (71) inflate the attack surface and MECM licensing counts.

F-07 · Firewall posture High

Four servers expose inbound Any-Any rules created during past troubleshooting (rule name: "temp - open all" on two of them). These bypass the segmentation model entirely.
